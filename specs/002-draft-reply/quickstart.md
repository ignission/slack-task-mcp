# Quickstart: draft_reply

**Feature**: 002-draft-reply
**Date**: 2025-12-18

## 概要

`draft_reply` は、Slackへの返信の下書きを添削し、ロジカルに構造化するMCPツールです：

- 結論→根拠→アクションの構造に整理
- Before/After形式で変更点を表示
- タスクタイプ別のテンプレート適用
- コピー用テキストを提供

## 基本的な使い方

### 1. 下書きを添削

```
ユーザー: この返信を添削して「レポートできました。添付します。確認お願いします。」
```

Claude が自動的に:
1. タスクタイプを判定（報告系）
2. 構造化（結論→根拠→アクション）
3. `draft_reply` で結果を整形

### 2. 出力例

```
## 添削結果

### Before
「レポートできました。添付します。確認お願いします。」

### After
「週次レポートが完成しましたので、添付いたします。
お手すきの際にご確認いただけますでしょうか。
ご不明点があればお知らせください。」

### 構造
- **結論**: 週次レポートが完成した
- **根拠**: （詳細は添付ファイル参照）
- **アクション**: 確認をお願いしたい

### 変更ポイント
1. **simplify**: 「できました→完成しました」に変更
   - 理由: より具体的で丁寧な表現に
2. **tone**: 「確認お願いします→ご確認いただけますでしょうか」
   - 理由: 相手の都合を考慮した依頼形に
3. **add**: 「ご不明点があれば」を追加
   - 理由: 次のアクションを明確にし、やり取りをスムーズに

---
📋 **コピー用**
週次レポートが完成しましたので、添付いたします。
お手すきの際にご確認いただけますでしょうか。
ご不明点があればお知らせください。
```

## ユースケース

### ケース1: 報告系（report）

**入力**:
```
「MTGの件、終わりました。特に問題なかったです。」
```

**出力**:
```
### After
「本日のミーティングが完了しましたのでご報告いたします。
議論は予定通り進行し、特に問題点はありませんでした。
議事録は本日中に共有いたします。」

### 構造
- **結論**: ミーティングが完了した
- **根拠**: 問題なく進行した
- **アクション**: 議事録を共有する
```

### ケース2: 確認系（confirm）

**入力**:
```
「明日の打ち合わせって10時からでしたっけ？」
```

**出力**:
```
### After
「明日の打ち合わせについて確認させてください。
開始時刻は10時からで間違いないでしょうか？
また、場所は会議室Aでよろしいですか？」

### 構造
- **結論**: 打ち合わせの時刻を確認したい
- **根拠**: 認識に齟齬がないようにしたい
- **アクション**: 時刻と場所の確認
```

### ケース3: 依頼系（request）

**入力**:
```
「資料のレビューお願いできますか」
```

**出力**:
```
### After
「お忙しいところ恐れ入りますが、添付の資料をご確認いただけますでしょうか。
来週月曜の会議で使用する予定です。
金曜日までにフィードバックいただけると助かります。」

### 構造
- **結論**: 資料のレビューをお願いしたい
- **根拠**: 来週の会議で使用するため
- **アクション**: 金曜日までにフィードバック
```

### ケース4: カジュアルトーン

**入力**:
```
ユーザー: カジュアルに添削して「了解です！やっておきます」
```

**出力**:
```
### After
「了解です！
今日中に対応しておきますね。
終わったら連絡します！」

### 変更ポイント
1. **add**: 具体的な期限（今日中）を追加
2. **add**: 完了連絡の約束を追加
```

## ワークフロー全体

```
1. ユーザー: 「このスレッドを分析して [URL]」
   ↓
2. Claude: get_slack_thread(url) → analyze_request
   ↓
3. ユーザー: 「この返信を添削して『〇〇』」
   ↓
4. Claude: draft_reply(draft_text, edited_reply) で結果を構造化
   ↓
5. ユーザー: コピー用テキストをSlackに貼り付け
```

## オプション指定

### タスクタイプを明示

```
ユーザー: 報告として添削して「作業完了しました」
```

### トーンを指定

```
ユーザー: カジュアルに添削して「ありがとうございます」
```

### 文脈を考慮

```
ユーザー: さっきのスレッドの文脈で添削して「確認しました」
```

## エラーケース

### 下書きが空の場合

```
❌ 添削対象のテキストを入力してください。

例: 「この返信を添削して『〇〇の件、了解しました』」
```

### 下書きが長すぎる場合

```
❌ 下書きが長すぎます（最大2000文字）。

要点を絞って再度入力してください。
```

## Tips

### 効果的な使い方

1. **まず下書きを書く**: 完璧でなくてOK、思いついたことをそのまま
2. **タスクタイプを意識**: 報告/確認/依頼のどれか考えると整理しやすい
3. **コピー用をそのまま使う**: 添削結果はそのままSlackに貼れる

### よくあるパターン

| シチュエーション | 推奨タスクタイプ |
|-----------------|-----------------|
| 作業完了の報告 | report |
| 質問への回答 | report |
| 日程・内容の確認 | confirm |
| 作業の依頼 | request |
| 資料のレビュー依頼 | request |
