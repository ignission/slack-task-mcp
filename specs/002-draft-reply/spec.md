# Feature Specification: draft_reply

**Feature ID**: 002-draft-reply
**Date**: 2025-12-18
**Status**: Draft

## Overview

Slackへの返信メッセージの下書きを添削し、ロジカルに構造化するMCPツール。「結論→根拠→アクション」の形式に整理し、相手に伝わりやすい返信を生成する。

## Problem Statement

Slackで返信を書くとき、以下の課題がある：

- 何から書き始めればいいかわからない
- 書いたけど論理的に伝わるか不安
- 冗長になりがち、要点がぼやける
- タスクタイプ（報告/確認/依頼）ごとの適切な構造がわからない

## User Stories

### P1: 基本的な添削

```
As a ユーザー
I want to 下書きを添削してもらう
So that 相手に伝わりやすい返信を書ける
```

**Acceptance Criteria**:
- 下書きテキストを入力として受け取る
- 「結論→根拠→アクション」の構造に整理される
- Before/After形式で変更点がわかる

### P2: タスクタイプ別テンプレート

```
As a ユーザー
I want to タスクタイプに応じた構造を選べる
So that 状況に適した返信形式で書ける
```

**Acceptance Criteria**:
- 報告系/確認系/依頼系のタイプを指定できる
- タイプに応じたテンプレート構造が適用される
- タイプを指定しない場合は自動判定

### P3: 改善ポイントの提示

```
As a ユーザー
I want to 具体的な改善ポイントを知りたい
So that 次回から自分で書けるようになる
```

**Acceptance Criteria**:
- 変更箇所と理由が明示される
- 論理の飛躍や抜け漏れがあれば指摘される
- 良い点があれば残す理由も説明される

### P4: 文脈を考慮した添削

```
As a ユーザー
I want to スレッドの文脈を考慮した添削がほしい
So that 会話の流れに沿った返信ができる
```

**Acceptance Criteria**:
- analyze_request の結果を参照できる
- スレッドの相手・トーンを考慮した表現
- 前の会話を踏まえた返信になる

## Functional Requirements

### FR-001: 下書き入力
- 添削対象のテキストを必須入力として受け取る
- 空文字・空白のみは無効

### FR-002: タスクタイプ指定
- report（報告）/ confirm（確認）/ request（依頼）を選択可能
- 省略時はClaudeが自動判定

### FR-003: 構造化出力
- 結論（conclusion）: 何を伝えたいか
- 根拠（reasoning）: なぜそう言えるか（省略可）
- アクション（action）: 何をしてほしいか / 自分が何をするか

### FR-004: Before/After表示
- 元の下書き（before）を表示
- 添削後のテキスト（after）を表示

### FR-005: 改善ポイント
- 変更した箇所と理由のリスト
- 論理チェック結果（飛躍・抜け漏れの指摘）

### FR-006: 文脈参照（オプション）
- thread_content を渡すとスレッド文脈を考慮
- 相手の名前や過去の発言を反映

### FR-007: トーン調整
- formal（丁寧）/ casual（カジュアル）を選択可能
- デフォルトは formal

### FR-008: コピー用テキスト
- 添削結果をそのままSlackに貼れる形式で出力

## Non-Functional Requirements

### NFR-001: レスポンス
- 5秒以内に結果を返す

### NFR-002: 文字数
- 入力: 最大2000文字
- 出力: 入力の2倍以下に収める

## Edge Cases

### EC-001: 既に構造化されている
- 大きな変更不要と判断し、微調整のみ行う
- 「良い構造です」とフィードバック

### EC-002: 文脈が不足
- thread_content なしでも動作
- 汎用的な添削を行う

### EC-003: 感情的な内容
- 攻撃的な表現は中立的に変換
- 変更理由を明示

## Out of Scope

- 自動送信機能
- 複数言語対応（日本語のみ）
- 音声入力対応

## Success Criteria

1. 下書きを入力すると構造化された返信案が得られる
2. Before/After形式で変更点が明確
3. タスクタイプに応じたテンプレートが適用される
4. 改善ポイントが具体的に提示される
5. 出力をそのままSlackに貼り付けられる

## Dependencies

- analyze_request（文脈参照時）
- CLAUDE.md の「返信添削のルール」
