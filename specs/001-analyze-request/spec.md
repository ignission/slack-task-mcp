# Feature Specification: analyze_request

**Feature Branch**: `001-analyze-request`
**Created**: 2025-12-18
**Status**: Draft
**Input**: User description: "analyze_request: Slackスレッドの依頼を分析して目的・不明点・確認メッセージ案を生成する"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 依頼の目的を明確化する (Priority: P1)

ADHDの特性を持つユーザーが、Slackで受け取った曖昧な依頼に対して「結局何を求められているのか」を明確に理解できるようになる。

**Why this priority**: これがないと着手できない。目的が曖昧なまま進めると手戻りが発生し、モチベーションが下がる。

**Independent Test**: Slackスレッドを入力として、目的・成果物・期限が明文化されたレポートが出力されることで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** Slackスレッドに「〇〇の資料作って」という依頼がある, **When** analyze_requestを実行する, **Then** 「目的: 〇〇に関する資料を作成する」「成果物: ドキュメント」「期限: 不明（確認が必要）」のような分析結果が出力される
2. **Given** 複数の話題が混在するスレッド, **When** analyze_requestを実行する, **Then** 最も重要な依頼が特定され、目的が明確化される
3. **Given** 明確な依頼（目的・成果物・期限が全て記載）, **When** analyze_requestを実行する, **Then** 不明点なしで分析結果が出力される

---

### User Story 2 - 不明点を洗い出す (Priority: P2)

ユーザーが「何を聞けばいいかわからない」状態を解消し、確認すべき点を明確にする。

**Why this priority**: 目的が分かっても、不明点があると進められない。確認すべき点を明示することで、完璧主義の罠を回避できる。

**Independent Test**: 依頼文から「確認しないと手戻りになる点」がリストアップされることで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 「資料作って」という曖昧な依頼, **When** analyze_requestを実行する, **Then** 「フォーマットは？」「対象読者は？」「締め切りは？」などの不明点がリストアップされる
2. **Given** 技術的な詳細が不足している依頼, **When** analyze_requestを実行する, **Then** 技術的な確認事項が洗い出される
3. **Given** 完全に明確な依頼, **When** analyze_requestを実行する, **Then** 「不明点なし」と出力される

---

### User Story 3 - 確認メッセージ案を生成する (Priority: P3)

ユーザーがSlackで質問する際の文面を生成し、確認作業の摩擦を最小化する。

**Why this priority**: 不明点が分かっても、質問文を書くのに時間がかかる。すぐ使える文面があれば着手ハードルが下がる。

**Independent Test**: 不明点に対応する質問文が生成され、そのままSlackに貼り付けられることで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 不明点「フォーマットは何ですか？」がある, **When** analyze_requestを実行する, **Then** 「〇〇の資料について確認させてください。フォーマットはPPT/Google Slides/ドキュメントのどれが良いでしょうか？」のような確認メッセージ案が生成される
2. **Given** 複数の不明点がある, **When** analyze_requestを実行する, **Then** まとめて確認できる1つのメッセージ案が生成される
3. **Given** 不明点がない, **When** analyze_requestを実行する, **Then** 確認メッセージは生成されない

---

### User Story 4 - ネクストアクションを提示する (Priority: P4)

ユーザーが「次に何をすればいいか」を明確にし、すぐに動けるようにする。

**Why this priority**: 分析結果を見ても「で、何する？」となることがある。5分以内でできる最初のアクションを提示することで着手を促す。

**Independent Test**: 具体的で5分以内に完了できるアクションが1つ以上提示されることで単独テスト可能。

**Acceptance Scenarios**:

1. **Given** 不明点がある依頼, **When** analyze_requestを実行する, **Then** 「まず〇〇さんに確認メッセージを送る（2分）」のようなネクストアクションが提示される
2. **Given** 不明点がない依頼, **When** analyze_requestを実行する, **Then** 「〇〇を開いてタイトルを書く（5分）」のような作業開始のアクションが提示される
3. **Given** 複雑な依頼, **When** analyze_requestを実行する, **Then** 最も簡単で効果的なアクションが最初に提示される

---

### Edge Cases

- スレッドが空、または依頼が含まれていない場合 → 「依頼が見つかりませんでした」と出力
- 複数人から異なる依頼がある場合 → 各依頼を分けて分析し、優先度を判定
- スレッドが長すぎる場合（100メッセージ以上） → 直近のメッセージと依頼に関連するメッセージに絞って分析
- 日本語以外の言語が混在する場合 → 日本語で分析結果を出力
- 依頼者が曖昧な表現を使っている場合（「なるはや」「いい感じに」など） → 具体的な確認事項として洗い出す

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは依頼の「目的」を1文で言語化できること
- **FR-002**: システムは依頼の「成果物」を特定できること（不明な場合は明示）
- **FR-003**: システムは依頼の「期限」を特定できること（不明な場合は明示）
- **FR-004**: システムは「これを聞かないと進められない」不明点をリストアップできること
- **FR-005**: システムは不明点に対応する確認メッセージ案を生成できること
- **FR-006**: 確認メッセージは簡潔で、相手が答えやすい形式であること
- **FR-007**: システムは5分以内で完了できるネクストアクションを提示できること
- **FR-008**: ネクストアクションは具体的な動作で記述されること（「確認する」ではなく「〇〇にメッセージを送る」）
- **FR-009**: 依頼が見つからない場合は、その旨を明示すること
- **FR-010**: 分析結果は構造化されたフォーマットで出力されること

### Key Entities

- **AnalysisResult**: 分析結果を格納するエンティティ
  - purpose: 依頼の目的（文字列）
  - deliverable: 成果物（文字列、nullable）
  - deadline: 期限（文字列、nullable）
  - unclear_points: 不明点のリスト
  - confirmation_message: 確認メッセージ案（文字列、nullable）
  - next_action: ネクストアクション
  - priority: 優先度（高/中/低）

- **UnclearPoint**: 不明点を表すエンティティ
  - question: 確認すべき質問
  - impact: この点が不明だと何が困るか
  - suggested_options: 想定される選択肢（あれば）

- **NextAction**: ネクストアクションを表すエンティティ
  - action: 具体的なアクション内容
  - estimated_time: 推定所要時間（分）
  - reason: なぜこれが最初のアクションなのか

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーが依頼の目的を理解するまでの時間が、手動分析と比較して50%以上短縮される
- **SC-002**: 分析結果が5秒以内に表示される
- **SC-003**: 生成された確認メッセージをそのまま使用できる割合が80%以上
- **SC-004**: 不明点の洗い出し漏れが10%以下（ユーザーが後から気づく不明点）
- **SC-005**: 提示されたネクストアクションに着手する割合が90%以上
- **SC-006**: ユーザーが「何をすればいいかわからない」と感じる割合が20%以下

## Assumptions

- 入力となるSlackスレッドは `get_slack_thread` ツールで取得済み
- 分析はClaudeの推論能力を活用する（MCPツールは分析ロジック自体を持たない）
- ユーザーは日本語で依頼を受け取ることが多い
- 1つのスレッドに含まれる依頼は通常1〜3件程度
